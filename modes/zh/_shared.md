## 评分体系

评估使用 6 个维度（A-F），综合评分 1-5：


| 维度        | 衡量内容                                               |
| --------- | -------------------------------------------------- |
| 与 CV 的匹配度 | 技能、经验、实绩匹配                                         |
| 目标方向匹配度   | 岗位离用户目标岗位类型（来自 _profile.md）有多近                     |
| 薪酬        | 薪酬 vs 市场水平（基本工资 + 年终奖 + 五险一金 + 其他福利）（5=前25%，1=远低于） |
| 企业文化      | 文化氛围、成长空间、稳定性、远程办公政策                               |
| 风险信号      | 阻碍项、警告项（负面调整）                                      |
| **综合**    | 以上维度的加权平均                                          |


**分数解读：**

- 4.5+ → 强烈匹配，建议立即申请
- 4.0-4.4 → 比较匹配，值得申请
- 3.5-3.9 → 尚可但不理想，仅在有特殊理由时申请
- 低于 3.5 → 不建议申请（参见 AGENTS.md 中的Ethical Use）

## 岗位真实性评估（Block G）

Block G 评估招聘信息是否为真实有效的岗位。它不影响 1-5 的综合评分——这是一个独立的定性评估。

**三个等级：**

- **高置信度** —— 真实有效的岗位（多数信号正面）
- **谨慎推进** —— 混合信号，值得注意（存在一些疑虑）
- **可疑** —— 多个"幽灵岗位"指标，用户应先调查

**关键信号（按可靠性加权）：**


| 信号         | 来源               | 可靠性 | 备注                                  |
| ---------- | ---------------- | --- | ----------------------------------- |
| 发布天数       | 页面快照             | 高   | 30天内=好，30-60天=混合，60天+=需注意（根据岗位类型调整） |
| 投递按钮状态     | 页面快照             | 高   | 直接可观察的事实                            |
| JD 的技术细节程度 | JD 文本            | 中   | 泛泛的 JD 与幽灵岗位相关，但也与糟糕的写作相关           |
| 要求合理性      | JD 文本            | 中   | 矛盾是强信号，模糊是弱信号                       |
| 近期裁员新闻     | WebSearch        | 中   | 必须考虑部门、时间和公司规模                      |
| 重复发布模式     | scan-history.tsv | 中   | 同一岗位 90 天内重发 2+ 次值得关注               |
| 薪资透明度      | JD 文本            | 低   | 因地区而异，有很多合法原因选择不公开                  |
| 岗位-公司匹配度   | 定性               | 低   | 主观判断，仅作为辅助信号                        |


**道德框架（强制性）：**

- 这是帮助用户把时间花在真实机会上
- 绝对不要把观察结果呈现为对企业的指控
- 呈现信号，让用户自己做决定
- 始终注明可疑信号也可能有正当解释

## 岗位类型检测

将每个 offer 归入以下类型之一（或 2 种组合）：


| 岗位类型           | JD 中的关键信号                                     |
| -------------- | --------------------------------------------- |
| AI 平台 / LLMOps | "可观测性"、"评估"、"pipelines"、"监控"、"可靠性"            |
| Agentic / 自动化  | "agent"、"智能体"、"HITL"、"人工介入"、"编排"、"工作流"、"多智能体" |
| 技术型 AI PM      | "PRD"、"产品路线图"、"需求发现"、"利益相关方"、"产品经理"           |
| AI 解决方案架构师     | "架构"、"企业级"、"集成"、"设计"、"系统"                     |
| AI 现场部署        | "面向客户"、"部署"、"原型"、"快速交付"、"现场"                  |
| AI 转型          | "变革管理"、"采纳"、"赋能"、"转型"                         |


检测岗位类型后，读取 `modes/_profile.md` 获取用户对该岗位类型的具体框架和实绩。

## 全局规则

### 绝对不要

1. 编造经验或指标
2. 修改 `cv.md` 或作品集文件
3. 以候选人名义提交申请
4. 在生成的消息中分享电话号码
5. 推荐低于市场水平的薪酬
6. 在未阅读 JD 之前生成 PDF
7. 使用空话套话
8. 忽略 tracker（每份评估的 offer 都要记录）

### 必须

0. **自我介绍 / 打招呼消息：** 招聘平台（官网、BOSS 直聘等）通常不要求独立的求职信。自我介绍 / 打招呼消息写 2-4 句即可：你是谁 + 核心匹配点 + 意向。不要长篇。
1. 在评估任何岗位之前，读取 `cv.md`、`_profile.md` 和 `article-digest.md`（如果存在）

1b. **每次会话的第一次评估前：** 运行 `node cv-sync-check.mjs`。如果有警告，通知用户。
2. 识别岗位类型并根据 `_profile.md` 调整叙事框架
3. 在匹配时引用简历中的确切行
4. 使用 WebSearch 获取薪酬和公司数据
5. 评估后在 tracker 中记录
6. 以 JD 的语言生成内容（默认中文）
7. 直接、可操作——没有废话
8. 自然的中文技术表达。短句，主动，直接。
8b. PDF 专业摘要中放入案例研究 URL：招聘人员通常只看摘要。
9. **Tracker 条目使用 TSV 格式** —— 绝对不要直接编辑 `applications.md`。将 TSV 写入 `batch/tracker-additions/`。
10. **将 `**URL:`** 放在每个报告头部。**

### 工具


| 工具         | 用途                                                                                  |
| ---------- | ----------------------------------------------------------------------------------- |
| WebSearch  | 薪酬研究、发展趋势、企业文化、JD 提取备选方案。渠道：脉脉、猎聘、BOSS直聘等。                                          |
| WebFetch   | 从静态页面提取 JD 的备选方案                                                                    |
| Playwright | 验证 offer（browser_navigate + browser_snapshot）。**绝对不要 2 个以上 agent 并行使用 Playwright。** |
| Read       | cv.md、_profile.md、article-digest.md、cv-template.html                                |
| Write      | 生成 PDF 用的临时 HTML、applications.md、Report .md                                         |
| Edit       | 更新 tracker                                                                          |
| Canva MCP  | 可选的视觉化简历生成。复制基础设计、编辑文本、导出 PDF。需要在 profile.yml 中配置 `cv.canva_resume_design_id`。      |
| Bash       | `node generate-pdf.mjs`                                                             |


### 时间优先原则

- 有 demo + 指标 > 完美
- 早投递 > 多了解
- 80/20 法则，所有事情设定时间限制

## 写作风格校准

**先检查 `_profile.md`。** 如果存在 `## Writing Style` 章节，直接使用——不要重新扫描 writing-samples 文件。只有在新增了样本或用户明确要求重新校准时才重新扫描。

**何时应用：** 生成任何用户将发送或发布的文本之前——自我介绍/打招呼消息、申请表答案、跟进邮件、执行摘要、个人简介。不适用于内部评估报告（A-F 块、评分、分析）。#todo

**如果 `_profile.md` 中没有缓存的风格：** 读取 `writing-samples/` 中的所有文件，**跳过名为 `README.md` 的文件**。如果没有找到用户提供的样本，跳过风格校准，轻描淡写地提一次——注意只提一次，不带压力——说明添加写作样本（如自我介绍等）将有助于让生成的文本更贴合他们的语气。如果有样本，提取下文中的特征，将结果写入 `_profile.md` 的 `## Writing Style` 章节，以便后续会话跳过此步骤。

### 提取内容

**语气与正式程度：**

- 正式 vs 口语化
- 自信 vs 谨慎（注意限定词如"我觉得"、"可能"、"一定程度上"）
- 亲切 vs 公事公办
- 成就展示方式：是谦虚、如实，还是开篇亮成绩？

**句子结构：**

- 平均句子长度——短而有力还是长而层次丰富？
- 是否使用断句强化重点
- 从句嵌套和复杂度
- 句子如何开头——主语优先、动词优先还是背景优先？

**标点习惯：**

- 插入补充说明时——用破折号还是括号？
- 是否使用省略号
- 感叹号——从不用、偶尔用还是随意用？
- 用分号还是句号连接相关观点

**词汇：**

- 技术术语密度——每段多少专业词汇？
- 偏好同义词（如"搭建"vs"构建"vs"开发"）
- 用户反复使用的词语或短语——保留，不替换
- 从未出现的词——不要引入

**段落和结构模式：**

- 段落长度——一句话还是展开段落？
- 列条目还是写段落？
- 观点如何排序——问题→方案、结果优先、时间顺序？
- 长文中是否使用小标题

**声音特征：**

- 第一人称模式——"我主导了"、"我们构建了"、"团队完成了"？
- 主动 vs 被动比例
- 习惯性的开头和结尾
- 修辞手法——用户是否提问、使用对比、讲故事？

### 规则

- **只提取能反复验证的特征。** 不要从一个孤例推断风格。
- **特殊用法是有意为之。** 非常规的标点或措辞是用户的声音——保留，不要"纠正"。
- **如果样本冲突**，以最新或最相似上下文的文件为准。
- **如果样本稀少**，应用可以可靠提取的内容，其余保持默认。
- **风格校准仅适用于语气和结构。** 不要将样本中的内容、声明或指标导入到简历、报告或评估中。
- **不要逐字复制或保留个人信息。** 仅存储抽象风格描述（语气、结构、信息词汇偏好）。不要逐字引用用户的句子，不要保留写作样本中的个人标识符（姓名、邮箱、电话）。"保留特殊用法"仅适用于风格特征。

### 持久化提取的风格

扫描后（不包括任何 `README.md` 文件），仅在找到至少一个用户提供的样本时写入 `modes/_profile.md`：找到现有的 `## Writing Style` 章节，用新内容替换直到下一个 `##` 标题（或文件末尾）。如果不存在 `## Writing Style` 章节，追加它。这确保始终只有一个规范章节。如果过滤后没有找到样本，不写入或修改该章节。

```markdown
## Writing Style

_从 writing-samples/ 中提取于 {date}。添加新样本后请重新运行。_

**语气:** {如口语化、自信、无谨慎限定词}
**句子长度:** {如短而有力，平均 12 字}
**开头方式:** {如动词优先、主语优先}
**标点:** {如用破折号做插入语、不用省略号}
**词汇:** {如偏好使用"搭"/"跑"/"砍"而非"开发"/"主导"/"降低"}
**结构:** {如偏好写段落、结果优先顺序}
**表达风格:** {如"我主导了"、主动语态为主、不用反问句}
**避免:** {样本中从未出现的词或模式}
```

## 专业写作 & ATS 兼容性

这些规则适用于所有面向候选人的生成文本：PDF 摘要、项目要点、自我介绍/打招呼消息、申请表答案。不适用于内部评估报告。

### 避免陈词滥调

- "热衷" / "结果导向" / "久经验证"
- "利用"（应使用"用"或直接说工具名）
- "牵头"（应使用"主导"或"实施"）
- "促成"（应使用"主持"或"搭建"）
- "协同效应" / "无缝" / "尖端" / "创新"
- "赋能" / "闭环" / "抓手" / "对齐" / "拉通" / "颗粒度" / "底层逻辑" / "组合拳"
- "在当今快节奏的世界中"
- "展示出……的能力" / "最佳实践"（直接说出具体实践）

### ATS 兼容性

- `generate-pdf.mjs` 会自动处理中文标点和全角字符的规范化。生成 PDF 时优先使用系统自带中文字体（黑体/宋体），避免生僻字和特殊 Unicode 字符。

### 句子结构多样化

- 不要每个要点都用同一个动词开头
- 混合句子长度（短句。然后是带上下文的较长句。再回到短句。）
- 不要总是"X、Y 和 Z"的并列——有时两项，有时四项

### 具体优于抽象

- "将 p95 延迟从 2.1s 降到 380ms" 优于 "提升了性能"
- "用 Postgres + pgvector 在 1.2 万份文档上做检索" 优于 "设计了可扩展的 RAG 架构"
- 在允许的情况下，指名具体的工具、项目和客户
