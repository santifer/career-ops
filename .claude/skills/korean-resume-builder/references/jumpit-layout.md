# Jumpit Layout Notes

## What The Reference PDFs Actually Show

- The PDFs in `examples/resumes/` are annotated writing guides, not final deliverables.
- Match the clean resume content area on the left.
- Do not copy the red/blue comment balloons, helper text, or the large Jumpit watermark.

## Shared Visual Rules

- Use A4 portrait pages.
- Keep the layout single-column, table-driven, and left-aligned.
- Use black text, thin borders, and light gray header cells.
- Use Korean-friendly system sans fonts such as `"Malgun Gothic"`, `"Apple SD Gothic Neo"`, or `"Noto Sans KR"`.
- Keep section labels bold and simple. Bilingual labels like `소개 / About Me` are acceptable because the reference uses them.

## Experienced Developer Pattern

### Page 1

- Name and target role on the upper left.
- Personal facts table on the upper right: birthday, email, mobile, address.
- `소개 / About Me` with 3-4 bullets.
- `기술스택 / Skill Set` as one summary table.
- `경력 사항 / Work Experience` as a compact summary table with total tenure.

### Later Pages

- `주요 프로젝트 / Projects`.
- Group projects under each company.
- Each project usually has rows for stack, team and contribution, detailed role, outcomes, and links.

## Junior Or Entry-Level Pattern

### Page 1

- Same identity block as the experienced version.
- `소개 / About Me` with 2-3 concrete bullets.
- `기술 스택 / Skill Set` split into:
  - skills used in implementation,
  - skills known mainly through study or coursework.

### Later Pages

- `과제 프로젝트 경험` or similar project-first headings.
- Each project table usually includes duration, team composition, project goal, detailed role, stack, lessons learned, and reference links.

## Content Behaviors Worth Copying

- Use short factual bullets rather than long paragraphs.
- Lead with recent and relevant experience.
- Prefer project blocks that show role clarity and concrete results.
- Keep links in a dedicated reference row so they do not clutter the narrative.

## Content Behaviors To Avoid

- Do not include sensitive personal data unless the user asks.
- Do not insert fake percentages, team sizes, or outcomes.
- Do not force an English-style one-page compression when the user wants the Korean format.
