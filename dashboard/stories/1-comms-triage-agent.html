"AI-native workflows in production — not just prototypes"

Source: Cursor (Anysphere) — Forward Deployed Engineer · Humanize: 9% low

The inbound request stream for the Office of Cross-Google Engineering wasn't just noisy — it was structurally unmanageable. xGE supports roughly 1,000 of Google's most senior technical individual contributors: Principal Engineers, Distinguished Engineers, Google Fellows. The top 0.5% of a 180,000-person organization. Every hour of senior IC time lost to low-complexity communications triage isn't an abstract inefficiency. It's a retention signal, a frustration accumulation, and at that seniority tier, a cost event with eight-figure implications if it compounds into attrition. I needed to build a system that resolved the routine, escalated the genuinely complex, and did both without a human in the loop.

The architecture I landed on was a single Gemini agent running a three-prompt sequential chain: triage, then revise, then escalate. The sequencing mattered — not multi-agent orchestration with separate models handing off state, but a single-model design with three discrete prompts each doing a bounded job. The first prompt classified inbound traffic into Low, Medium, and High touch tiers against a conditional knowledge base that loaded context specific to the request category. That conditional KB loading was the design decision I'm most glad I didn't skip. A flat, monolithic knowledge base would have introduced irrelevant retrieval noise on every pass; tiered loading kept the model's working context clean and the classification sharper. The second prompt applied Smart Brevity formatting to the Low-touch tier — resolved with documented rationale so the IC could audit the resolution rather than accept a black box. The third prompt generated a structured escalation package for High-touch traffic, with enough classification metadata attached that a human reviewer could triage the escalation queue without re-reading the original thread from scratch.

The craft detail that doesn't appear in most agent postmortems: I built the rationale documentation into the Low-touch resolution layer deliberately, not as a compliance check but as an adoption mechanism. Senior engineers at the Principal and above tier don't accept opaque automation. They will route around a system they don't trust, and once they route around it, the classification accuracy data becomes useless because the input set is no longer representative. Documented rationale on every resolution kept the loop honest and kept the senior IC population inside the system rather than filing exceptions. That's not a prompt engineering decision — it's an organizational behavior decision expressed through prompt engineering.

The system recaptured approximately 160 operational hours per year at greater than 90% classification accuracy across the full inbound stream. Those aren't projected figures from a pilot — the agent was in production, running against real traffic from the actual population it was built for. The more telling proof point came later: during a multi-month medical leave in early 2026, the triage system maintained 100% operational continuity without a single human intervention. No coverage arrangement, no handoff document, no deputy. The system held. That's the difference between an AI-native workflow and a prototype with good demo metrics — a prototype doesn't run unattended through an extended absence and come back intact.

What I learned building this is that the hardest design problem in production agent work isn't the model — it's the organizational surface the agent is touching. The triage system worked because I understood the population it served well enough to build the right escalation thresholds, the right KB structure, and the right rationale format for that specific audience. The technical primitives — prompt chaining, conditional retrieval, structured output — are learnable by anyone in a weekend. Knowing which senior IC request patterns genuinely require human judgment versus which only feel like they do because no one has ever systematically categorized them — that's the knowledge that doesn't come from the documentation.
