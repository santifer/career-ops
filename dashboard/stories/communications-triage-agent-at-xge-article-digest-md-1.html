Review technical architectures + production-ready code

Source: Anthropic — Manager, Forward Deployed Engineering · Humanize: 9% low

The inbound stream for the Office of Cross-Google Engineering was not designed to be a problem. It became one the way most operational problems do — gradually, then all at once. xGE is a horizontal engineering organization supporting roughly 1,000 Principal, Distinguished, and Fellow engineers, the top 0.5% of Google's 180,000-person workforce. When that population sends communications requests, the stakes attached to each one are not hypothetical. A misclassified ask from a Google Fellow, a slow turnaround on a VP-level escalation, a templated response where a calibrated one was required — these are not minor inconveniences. They erode trust in the exact cohort where trust is most expensive to rebuild. The triage layer we had was human. That meant it was me. And that wasn't going to scale.

The architectural decision that mattered most came early, and it wasn't about the model. It was about whether to build a multi-agent network or a single agent with a deliberately constrained prompt chain. Multi-agent systems carry real operational appeal — parallelism, specialization, apparent sophistication. They also carry compounding failure surfaces. For this use case, the population was defined, the inbound categories were finite, and the failure mode I was most concerned about was misclassification at the routing layer — a High-touch request treated as Low and resolved without escalation. That failure is invisible until it isn't. I chose a three-prompt architecture running inside a single Gemini agent: triage, revise, escalate. Each prompt stage had a conditional KB load — the agent pulled relevant context only when the classification warranted it, rather than flooding every request with the full corpus. Low-complexity inbound got Smart Brevity treatment with documented rationale appended so any human reviewer could reconstruct why the agent resolved rather than escalated. Medium and High tiers routed to escalation with a classification memo attached. The three-stage chain created an internal audit trail. Every resolved request was arguable — you could trace the logic backward from the output.

The production deployment question was the one I cared about most, because prototypes that solve the demo are not the same thing as systems that hold. The test I set for this agent was not whether it performed well under supervised conditions. It was whether it would maintain operational continuity during an uncontrolled absence — a real one, not a tabletop exercise. That test ran during a multi-month medical leave in early 2026. The agent handled the inbound stream without a single human intervention across the full absence window. 100% operational continuity. That result is the one I'd point to if someone asked me whether the architecture was production-ready, because no prototype can claim it. Only a deployed system under real load, with no human in the loop, can.

The measurable output: approximately 160 operational hours per year recaptured at better than 90% classification accuracy across the full inbound stream. For a cohort where each percentage point of senior IC attrition carries estimated eight-figure cost implications, the hours recaptured are almost secondary. The primary result is that the triage layer no longer has a single point of failure. It runs unattended. The system's ability to hold through a multi-month absence is the architectural validation — not the classification rate, which is a quality metric, but the continuity rate, which is an infrastructure metric. Those are different claims, and both are documented.

What this project clarified for me is the distinction between reviewing an architecture and owning one. I've read enough system design documentation to know that a three-prompt chain sounds simple on paper. The complexity is in the conditional KB loading logic, in the classification boundary definitions between tiers, in the documented rationale requirement that keeps the system auditable rather than opaque. Reviewing production-ready code means knowing which decisions are load-bearing and which are cosmetic — and being honest about the difference. This system is small by most engineering standards. It is also running, unattended, for the most senior technical population at one of the largest engineering organizations in the world. That's the bar I built to.
