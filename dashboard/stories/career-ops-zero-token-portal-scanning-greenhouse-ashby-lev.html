Automated end-to-end workflows

Source: Perplexity — Member of Technical Staff (Forward Deployed Engineer, Applied AI) · Humanize: 2% low

The job search was eating hours I didn't have. Every application required checking three separate portals — Greenhouse, Ashby, Lever — manually, repeatedly, on no consistent schedule, with no memory of what I'd already seen. The signal-to-noise ratio was brutal. I needed a system that ran without me.

So I built one.

The core design constraint was zero-token portal scanning. Greenhouse, Ashby, and Lever all expose API endpoints that return structured job data — and critically, they return it before you ever touch an LLM. That distinction mattered. Most job-search automation I'd seen burned tokens on every pass, running every listing through a language model regardless of whether the role was relevant. That's the wrong architecture — it's expensive, it's slow, and it means the expensive reasoning layer is doing work that structured data filtering should handle first. I rewrote the ingestion layer to hit the portal APIs directly, parse the structured JSON responses, and apply rule-based filters against my role criteria before a single token was spent. The LLM only sees listings that already passed the filter. Everything upstream of it is deterministic and cheap.

The parallel-worker layer came next. Sequential processing against three portals with overlapping posting schedules creates a bottleneck — you're waiting on each portal's response before starting the next. I built the workers to run concurrently, each handling its own portal scan and returning structured output to a single Claude orchestrator that handles the synthesis and downstream decisions. The orchestrator delegates; it doesn't do the fetching itself. That separation of concerns — orchestrator versus workers — is the same architectural instinct I used in the communications triage agent at Google's Office of Cross-Google Engineering, where a three-prompt routing architecture kept the decision logic clean and the operational costs predictable. Same principle, different domain. The orchestrator reasons; the workers collect.

The scheduling layer is what made it autonomous in the way that actually matters — unattended. I deployed it on launchd, macOS's native daemon scheduler, so the pipeline runs on a set interval without my involvement. I don't open a terminal. I don't trigger it manually. It runs, scans, filters, scores, and surfaces. I check the output. That's the entire interaction model. The production deployment went live in April 2026; the public repo is at github.com/mitwilli-create/career-ops, forked from the open-source career-ops project and extended with my personal corpus integration — voice profile, role files, project artifacts — so the downstream generation layer has full context when it does fire.

The measurable result is time architecture. I can't give you a before/after hour count the way I can with the 160 operational hours recaptured by the xGE triage agent, because the job search is a different kind of work — but the structural outcome is precise: a process that previously required active attention on an irregular, anxiety-driven schedule now runs on a fixed cadence I never think about. The cognitive overhead of "did I check Lever today" dropped to zero. The pipeline is a cron job that happens to understand my hiring criteria.

What I find worth naming here is that this is the same discipline that runs through every system I've shipped — the instinct to ask what should be automated out of human attention entirely, versus what requires human judgment and should be protected from automation. Portal scanning is the former. Deciding whether a role fits my actual interests is the latter. The system doesn't make that call. It just makes sure I never miss the input I need to make it myself.
