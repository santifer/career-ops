"Build full-stack systems that deliver customer value"

Source: OpenAI — Forward Deployed Engineer (FDE) - Seattle · Humanize: 2% low

The career-ops fork started as a personal irritant. I was running a serious AI job search while simultaneously doing the work that qualified me for it — building production agents at Google — and the tooling available for that search was almost comically mismatched with the kind of work I was actually pitching. Generic job-board scrapers. Prompt templates that read like everyone else's prompt templates. No architecture, just noise. So I built the infrastructure instead.

The foundation was the open-source `career-ops` project, which I forked and extended with a corpus integration layer I hadn't seen done anywhere: a personal voice profile derived from an analysis of roughly 1.08 million words of my own writing across Gmail, LinkedIn, Instagram, Facebook, and iMessage going back to 2005. The Voice OS project — which I'd built separately — had already surfaced my specific voice signatures, my banned phrases, and my AI-detection risk patterns, including the finding that my professional emails had grown from 289 to 853 words over time, making length the single biggest variable I could control for AI-detection exposure. That corpus became the calibration layer for everything the pipeline produced. When the system generated cover letters or story drafts, it wasn't approximating a generic professional register — it was working from a documented voice model with a kill list of constructions I actually reject.

The architecture ran a single Claude orchestrator with skill-based delegation across pipeline stages. I added portal scanning — zero-token hits against Greenhouse, Ashby, and Lever APIs — so the system could surface role matches without burning context window on full-page scrapes. Parallel worker batches handled the processing; the orchestrator handled prioritization and delegation. The build ran on Claude Code Skills, which meant I was writing agent logic in a framework I knew from my xGE work, not learning something new under deadline pressure. I shipped to production in April 2026 with the public repo at github.com/mitwilli-create/career-ops. The whole system is one Claude orchestrator doing what I'd already proven worked at scale — route, classify, resolve, escalate — except the principal it's serving is me, and the inbound stream is job descriptions instead of communications requests.

The result is a full-stack pipeline that actually closes the loop. It scans, matches, retrieves, generates, and calibrates output against a personal voice model — without a human in the loop for the first three stages. The triage agent I built for xGE recaptured roughly 160 operational hours per year at greater than 90% classification accuracy across a population of around 1,000 senior engineers. The career-ops fork applies the same architecture to a different domain: the overhead of a high-signal job search. It doesn't just find postings — it produces tailored, voice-matched artifacts that are ready to ship.

What I find most useful about this story for an FDE conversation is what it demonstrates about how I scope problems. I didn't build a job-search tool because I needed one — I built a full-stack system because the problem had a full-stack shape. Portal scanning is a different engineering problem from voice calibration, which is a different engineering problem from parallel-batch generation, which is a different engineering problem from orchestration. The instinct to see each of those as a system layer rather than a feature is the same instinct that built the Voice DNA RAG pipeline for VP-level communications and the autonomous triage agent that ran unattended through a multi-month absence without a single intervention failure. Customer value — whether the customer is a VP-level principal at Google or a senior engineer looking for their next role — doesn't come from the individual components. It comes from the architecture that connects them.
