Ambiguous-problem-to-agentic-spec translation

Source: Cohere — Applied AI Engineer – Agentic Workflows Humanize: 5% low

The problem didn't come with a requirements document. It came with a pattern I kept watching repeat itself: senior engineers at the Principal, Distinguished, and Fellow level — the top 0.5% of a 180,000-person organization — losing working time to inbound communications requests that had no triage layer between arrival and an actual human decision. The requests ranged from low-stakes informational routing to drafts that needed executive judgment before anything moved. Everything hit the same inbox. Everything waited the same amount of time. The cost of that undifferentiated queue, applied to that cohort, wasn't theoretical.

The first thing I did wasn't build anything. It was read the ambiguity carefully enough to find the actual problem underneath the stated one. The stated problem was volume. The real problem was classification — nothing in the existing workflow could tell the difference between a request that deserved two minutes of Smart Brevity and one that warranted VP visibility before a response went out. Once that distinction became clear, the architecture followed directly: not a chatbot, not a summarizer, but a three-stage routing system that could make that call autonomously and document the rationale for every decision it made.

I scoped the agent around a three-prompt architecture — triage, revise, escalate — with conditional KB loading that varied what context the agent pulled based on where in the routing tree a given request landed. A Low-touch inbound got Smart Brevity applied with a documented rationale trail. A Medium-touch request got revised against the relevant style and governance constraints before anything surfaced to a human. A High-touch flag stopped the autonomous loop and handed off cleanly with enough context that the human picking it up didn't have to reconstruct what had already happened. The conditional KB design was deliberate: I didn't want a system that loaded everything for every request, because that introduces noise at exactly the moment you need signal — when the triage decision is still forming. Loading context conditionally, based on early classification, kept the agent's reasoning path cleaner and made the escalation criteria legible when someone needed to audit a decision after the fact. That auditability wasn't a nice-to-have. This was a communications system running for Google Fellows. Every escalation decision needed to be reviewable.

What the architecture recaptured was approximately 160 operational hours per year at greater than 90% classification accuracy across the full inbound stream. Not a prototype — a production deployment serving the actual population it was designed for, running without a human in the loop on the resolved subset. The sharpest proof point came later, when the system maintained 100% operational continuity through a multi-month absence without a single intervention required. The queue didn't back up. The escalations went to the right people. The Low-touch responses went out with documented rationale attached. The agent held the line because the classification logic was sound, not because anyone was watching it.

What I learned from building this is the part that transfers. The hardest moment in any ambiguous-to-agentic translation isn't the prompt engineering — it's the earlier decision about where the autonomous loop should end and a human should re-enter. Get that boundary wrong and you either build a system that escalates everything and changes nothing, or one that resolves things it shouldn't and creates downstream damage that's hard to trace. The triage agent worked because I spent more time on that boundary than on any other design question. The architecture was a consequence of that clarity, not the other way around.
