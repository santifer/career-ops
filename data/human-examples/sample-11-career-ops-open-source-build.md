The career-ops project started as a practical problem: I was running a job search the same way I'd run everything else for the past two years — with agents doing the work I didn't need to do myself. But the open-source `career-ops` repository I forked as a starting point wasn't built for that. It was a scaffold. I extended it into a production pipeline.
What I built on top of that base was specific. I integrated my personal corpus — voice profile, role artifacts, project documentation, the full written record I'd already used to calibrate Voice DNA inside Google — directly into the pipeline so every generated output started from a documented, calibrated version of how I actually write rather than a generic approximation. The portal scanning layer used zero-token API hits against Greenhouse, Ashby, and Lever — no scraping, no rate-limit exposure, just direct reads against the endpoints those platforms expose. And the processing architecture ran parallel workers in batch rather than sequential single-job generation, with a single Claude orchestrator delegating across pipeline stages through skill-based handoffs rather than one monolithic prompt trying to do everything at once.
That architecture choice mattered for a specific reason. The failure mode I was trying to avoid wasn't quality — it was coherence at scale. When you're generating tailored materials across a large number of roles simultaneously, the risk isn't that any single output is bad. The risk is that the outputs drift from each other and from the source corpus in ways that are hard to catch without a review layer. The parallel-worker design with a single orchestrator kept the corpus integration centralized while letting the generation fan out. Claude Code was the right tool for this because it let me build and iterate on the skill layer — the modular components the orchestrator delegates to — incrementally, testing each stage as a discrete unit before wiring it into the full pipeline. I could pull a single skill out, test its output against the corpus, and push it back without touching the rest of the architecture. That's a different development experience than prompt chaining inside a chat interface, and the difference showed up in how quickly I could debug edge cases at the handoff points between stages.
The deployment went live in April 2026. The public repo is at github.com/mitwilli-create/career-ops. I also shipped a related calibration system — Voice OS — that analyzed roughly 1.08 million words of my own writing across Gmail, Instagram, Facebook, LinkedIn, and iMessage going back to 2005, surfacing voice signatures, banned phrases, and AI-detection risk patterns. That analysis fed directly into the corpus integration layer of the pipeline. One finding from that work: my professional emails grew from 289 to 853 words across the full trajectory, projecting toward 1,000-plus within 12 months — length turned out to be the single biggest AI-detection risk factor under my own control, and knowing that changed how I configured the output constraints for the generation stage.
What I take from building this — and from the triage agent and RAG pipeline I shipped inside Google — is that Claude Code specifically rewards a particular kind of discipline: you have to know what you're trying to test before you build the component that tests it. The tooling doesn't protect you from architectural confusion. It just removes friction from execution, which means unclear thinking ships faster and breaks harder. The projects where I've gotten the most out of Claude Code were the ones where I came in with the stage boundaries already drawn and used the skill architecture to enforce them, rather than discovering the structure during generation. That's not a limitation of the tool. It's the correct way to use it.
