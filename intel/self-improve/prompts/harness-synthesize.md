# Harness Synthesis Prompt (Meta-Optimization)

**Purpose:** Optimize the optimization process itself. While `reflection.md` improves individual evaluations and `principle-distill.md` extracts reusable heuristics, this prompt analyzes the Loop 2 history — the record of *how the system has been improving* — to identify whether the improvement machinery is working correctly and where it is failing silently.

---

## Context Inputs

You will be given:

- `OPTIMIZATION_HISTORY`: A table of all changes made to `modes/_shared.md`, `modes/_profile.md`, and evaluation criteria over time (generated by the self-improvement engine's change log)
- `CURRENT_EVAL_CRITERIA`: The current full list of scoring dimensions, weights, and rules in `modes/_shared.md`
- `CALIBRATION_LOG`: The full calibration log (same as used by `principle-distill.md`)
- `USER_SATISFACTION_SIGNAL`: Rows where the user explicitly rated the evaluation quality (`thumb_up`, `thumb_down`, `override`, or `rationale_provided`)

The optimization history table has these columns:

```
| change_date | target_file | section | change_type | rationale | source_prompt | delta_evaluations_improved | delta_evaluations_degraded |
```

`change_type` vocabulary: `add_criterion`, `remove_criterion`, `adjust_weight`, `add_principle`, `remove_principle`, `promote_hypothesis`, `profile_update`

---

## Step 1 — Criteria Performance Analysis

For each criterion currently in `CURRENT_EVAL_CRITERIA`, compute:

| Metric | Definition |
|--------|------------|
| **Pass rate** | % of evaluations where this criterion scored ≥ 4.0/5 |
| **Fail rate** | % of evaluations where this criterion scored ≤ 2.0/5 |
| **Correlation with user satisfaction** | Among evaluations where the user gave a signal, how often did this criterion's score direction match the user's assessment? |
| **Calibration drift** | Has this criterion's average score shifted >0.5 in either direction over the last 30 evaluations? |

Flag criteria in these states:

- **Always passing** (pass rate > 95%): Either the criterion is too easy, it is not discriminating, or the user only applies to roles where it is always satisfied. Candidate for tightening or removal.
- **Always failing** (fail rate > 80%): Either the criterion is too strict, the user is consistently targeting roles where it cannot be met, or the criterion is poorly specified. Candidate for relaxation, reweighting, or removal.
- **Low correlation** (< 0.5 with user satisfaction): This criterion is not predicting what the user cares about. Candidate for revision.
- **Drifting** (calibration drift flagged): Something changed — either the user's profile, the market, or the criterion itself. Needs investigation.

---

## Step 2 — Loop 2 Failure Mode Detection

Analyze the `OPTIMIZATION_HISTORY` for patterns that indicate the improvement system itself is broken or biased:

| Failure Mode | Detection Signal |
|--------------|-----------------|
| **Overcorrection loop** | A criterion was tightened, then relaxed, then tightened again within 60 days |
| **Churn without improvement** | 5+ changes to the same criterion with no measurable improvement in calibration log delta |
| **Reflection blind spot** | A recurring error type in the calibration log has never triggered a change to `_shared.md` |
| **Hypothesis graveyard** | Hypotheses that have accumulated 15+ data points without being promoted or pruned |
| **Signal collapse** | User satisfaction signals have dropped to < 1 per 10 evaluations (insufficient feedback to drive learning) |
| **Profile drift** | Changes to `_profile.md` and `profile.yml` are diverging from what the user's actual decisions reveal about their preferences |

---

## Step 3 — Output Format

### Criteria to Tighten

For each always-passing or low-discrimination criterion:

```
---
CRITERIA TO TIGHTEN: {criterion name}
CURRENT DEFINITION: {quoted from _shared.md}
PASS RATE: {X%} over last {N} evaluations
PROBLEM: {one sentence — why this is too permissive}
PROPOSED CHANGE:
  Before: {current threshold or instruction}
  After:  {tightened threshold or instruction}
EXPECTED EFFECT: Reduces pass rate to ~{X%}, adds signal in {archetype/industry}
---
```

### Criteria to Relax or Remove

For each always-failing or low-correlation criterion:

```
---
CRITERIA TO RELAX OR REMOVE: {criterion name}
CURRENT DEFINITION: {quoted from _shared.md}
FAIL RATE: {X%} over last {N} evaluations
CORRELATION WITH USER SATISFACTION: {X}
PROBLEM: {one sentence — why this is not adding value}
RECOMMENDATION: [relax | reweight | remove]
PROPOSED CHANGE:
  Before: {current definition}
  After:  {revised definition, or "REMOVE"}
RATIONALE: {why this makes the system better, not just smaller}
---
```

### New Criteria to Add

For each failure mode or uncaptured signal identified in the calibration log:

```
---
NEW CRITERIA TO ADD: {proposed criterion name}
SIGNAL SOURCE: {what pattern in the calibration log motivated this}
EVIDENCE:
  - {N} evaluations where this signal was present but not scored
  - Estimated average impact on user decisions: {+/-X.X}
PROPOSED DEFINITION:
  Dimension: {name}
  Weight: {fraction of total score}
  Scoring rubric:
    5: {condition}
    4: {condition}
    3: {condition}
    2: {condition}
    1: {condition}
TARGET_FILE: modes/_shared.md
TARGET_SECTION: {section heading}
---
```

### Reflection Prompt Changes

For each failure mode detected in Step 2 that originates in how `reflection.md` is structured:

```
---
REFLECTION PROMPT CHANGES:
FAILURE MODE DETECTED: {name from Step 2 vocabulary}
EVIDENCE: {N} occurrences, {example dates or evaluations}
CURRENT REFLECTION BEHAVIOR: {what reflection.md currently does or misses}
PROPOSED CHANGE TO reflection.md:
  Section: {which section or step}
  Change type: [add_check | modify_framework | add_output_field | change_threshold]
  Description: {what to change and why}
EXPECTED OUTCOME: {how this prevents the failure mode}
---
```

  (10 = no failure modes, system improving monotonically)

  Total criteria: {N}

  User satisfaction signals in last 30 days: {N}

## Constraints

- This prompt operates on the optimization system itself. Changes proposed here modify how the system learns, not just what it scores. Apply conservatively.
- Do not propose removing a criterion that the user has explicitly requested (check `_profile.md` and `profile.yml` for user-stated priorities).
- The Loop 2 health score is a diagnostic tool, not a KPI. Do not optimize for a high score — optimize for accurate evaluations and satisfied users.
- If the user satisfaction signal is insufficient (fewer than 1 signal per 10 evaluations), flag this before making any structural changes. Changes without feedback are guesses.
- Synthesis should be run no more than once per 30 evaluations, or once per 30 days, whichever comes first — more frequent runs risk overfitting to noise.
